# 认证与授权机制分析

## 1. 模块定位

认证与授权模块负责回答两个核心问题：

- 当前请求是谁发起的
- 当前用户是否有权限执行该操作

在本系统中，这两个问题并不是由单一框架完全接管，而是通过 `UserServiceImpl`、`JwtTokenManager`、`AuthCheck` 注解、`AuthInterceptor` 切面以及 Session / Redis 等机制共同实现。

因此，从论文角度看，本模块应被描述为“基于 JWT、Session 与角色校验相结合的轻量级认证授权机制”。

## 2. 整体实现结构

结合代码结构，认证与授权相关实现主要包括：

- `UserController`：提供注册、登录、退出、获取当前登录用户等入口
- `UserServiceImpl`：负责登录态写入、登录用户获取、退出清理等核心逻辑
- `JwtTokenManager`：负责 JWT 令牌生成、解析、失效与黑名单控制
- `JwtProperties`：维护令牌密钥、过期时间、请求头等配置
- `AuthCheck`：用于在方法上声明角色要求
- `AuthInterceptor`：通过 AOP 在方法执行前进行权限校验
- `UserRoleEnum`：定义系统角色枚举

这种结构体现出认证和授权已经被分层拆分，而不是全部堆积在控制器中完成。

## 3. 认证机制分析

### 3.1 登录入口

系统当前支持三种登录入口：

- 账号密码登录
- 手机号密码登录
- 微信开放平台登录

无论用户从哪一个入口进入，最终都会进入统一的登录态建立流程。

### 3.2 令牌生成

在用户登录成功后，`UserServiceImpl` 会调用 `JwtTokenManager.createToken(user)` 生成 JWT。当前令牌中主要包含以下载荷：

- `uid`：用户 ID
- `role`：用户角色
- `iat`：签发时间
- `exp`：过期时间

与此同时，系统还会把 token 返回到 `LoginUserVO` 中，并由控制器写入 HttpOnly Cookie。

### 3.3 Session 登录态

除了 JWT 之外，系统还会在 Session 中保存登录用户 ID，如：

- `user_login`
- `LOGIN_USER_ID`

从 `application.yml` 可见，系统启用了 Spring Session Redis，并配置了较长的会话超时时间。这说明系统采用的是“JWT + Session 并存”的登录态机制。

### 3.4 登录用户获取逻辑

`UserServiceImpl.getLoginUser()` 的处理逻辑非常典型：

1. 优先尝试从 JWT 中解析用户
2. 若 JWT 不存在或无效，则尝试从 Session 中获取用户
3. 若两者都无法获取，则视为未登录

这种顺序说明系统更偏向将 JWT 作为首选登录凭证，同时保留 Session 作为兼容与补充方案。

## 4. JWT 机制实现特点

`JwtTokenManager` 是认证链路中的关键组件，其核心能力包括：

- 创建令牌
- 从请求头或 Cookie 中解析令牌
- 校验令牌签名与过期时间
- 根据 token 反查用户 ID
- 令牌退出时加入黑名单

当前系统支持从以下位置读取 token：

- `Authorization` 请求头
- 名为 `token` 的自定义请求头
- 名为 `token` 的 Cookie

这意味着系统在接口调用方式上兼顾了浏览器环境和接口调试环境。

## 5. 退出与失效机制

如果系统只生成 JWT 而不支持失效控制，那么用户退出登录后，理论上旧令牌仍可能继续可用。当前系统通过 Redis 黑名单机制解决了这个问题。

具体流程为：

1. 用户调用退出接口
2. 系统解析当前 token
3. 将 token 的摘要写入 Redis 黑名单，并设置剩余 TTL
4. 后续每次解析 token 时，先检查是否命中黑名单

这种设计在单体应用中属于较常见且实用的 JWT 失效方案。

## 6. 授权机制分析

### 6.1 角色定义

系统当前定义了四类角色：

- `student`
- `teacher`
- `admin`
- `ban`

其中：

- `student` 是普通学习用户
- `teacher` 用于课程与教学内容管理
- `admin` 拥有全局管理权限
- `ban` 表示被封禁用户

### 6.2 声明式权限控制

在接口层，系统通过 `@AuthCheck(mustRole = "...")` 注解声明角色要求。例如：

- 管理员接口通常要求 `admin`
- 课程内容维护接口常要求 `admin,teacher`
- 未标注注解的接口则通常依赖业务层自行判断或允许公开访问

这种方式使授权规则与接口定义保持较高可读性。

### 6.3 AOP 权限校验

`AuthInterceptor` 会在方法执行前读取 `@AuthCheck` 配置，并完成如下校验：

1. 获取当前登录用户
2. 解析当前用户角色
3. 封禁用户直接拒绝访问
4. 管理员默认全局放行
5. 若注解要求多个角色，则按逗号分隔逐一匹配
6. 若角色不满足，则抛出无权限异常

从实现上看，这种方式属于典型的“注解 + AOP”声明式授权模型。

## 7. 认证与授权的协同关系

在本系统中，认证与授权不是彼此独立的两个点，而是连续的处理链：

`登录成功 -> 生成 token / 写入 session -> 请求到达接口 -> 解析当前用户 -> 判断角色 -> 进入具体业务逻辑`

进一步说：

- 认证解决“能否识别用户”
- 授权解决“能否执行操作”
- 业务层补充解决“能否操作这条具体数据”

例如，管理员可以全局放行，但教师管理课程时仍需要通过 `courseService.assertCanManageCourse(...)` 这类业务规则校验课程归属。

这说明系统不仅有角色级授权，也存在数据归属级授权。

## 8. 业务流程示例

以“教师修改课程章节”为例，可以将整条权限链概括为：

1. 教师登录成功，系统生成 token 并保存 Session
2. 教师请求受保护接口
3. `UserServiceImpl` 从 token 或 Session 中解析当前用户
4. `AuthInterceptor` 判断当前角色是否满足 `admin,teacher`
5. 若通过，则进入 Controller / Service
6. 业务层继续校验该教师是否真的拥有该课程的管理权限

这种多层校验方式能够避免仅靠角色判断带来的权限过宽问题。

## 9. 当前实现的工程特点

从工程角度看，本系统的认证授权实现具有以下特点：

- 不依赖完整的 Spring Security 权限体系，而是采用轻量自定义方案
- 同时兼容 JWT 和 Session 两种登录态来源
- 通过 Redis 黑名单解决 JWT 退出失效问题
- 通过注解与 AOP 让角色校验从业务代码中抽离
- 在角色授权之外保留业务归属判断，提升安全性

这类方案对于毕业设计和中小型单体系统来说，具有实现成本低、可读性较强、可直接落地的优点。

## 10. 可优化点

尽管当前实现已经较完整，但从论文分析角度仍可以指出一些可优化方向：

- 密码当前仍采用加盐摘要，后续可升级为更安全的密码散列算法
- JWT 与 Session 并存虽然兼容性较好，但也会提高登录态管理复杂度
- 角色模型当前较简单，若后续权限颗粒度继续细化，可进一步扩展资源级权限控制

这类说明有助于在论文中体现作者对系统安全性和可扩展性的思考。

## 11. 论文写作建议

在论文写作中，本专题建议重点突出以下关键词：

- 多方式登录
- JWT 与 Session 并存
- Redis 黑名单失效控制
- 注解式角色授权
- 角色授权与业务归属授权结合

这样既能够讲清认证授权机制的实现方式，也能体现系统具备一定工程规范意识。

## 12. 模块分析结论

综合来看，系统当前采用了以 JWT 为主、Session 为辅的认证机制，并通过 `@AuthCheck` 与 `AuthInterceptor` 构建了角色驱动的授权体系。与此同时，系统还在具体业务中补充了数据归属校验，从而形成了“身份识别、角色判断、业务权限控制”逐层递进的安全访问机制。

因此，在论文中可以将本模块概括为：**系统基于 JWT、Session、Redis 黑名单与 AOP 权限校验机制，实现了适用于教学平台场景的轻量级认证与授权方案。**
